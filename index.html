Scope

This style guide applies to all technical writers, engineers, subject matter experts (SMEs), and stakeholders who create, review, or approve customer-facing documentation.

EQ_DCC technical writers will adhere to a structured documentation approach where content is written as independent, self-contained topics that can stand alone and be reused. Each topic covers one clear objective: understanding a concept, performing a task, or referencing information.

Modular documentation directly reduces cognitive load because it:

Prevents information overload.

Supports a predictable structure (concept-task-reference).

Limits the mental effort required to locate relevant information.

Minimizes unnecessary reading.

Helps users learn or perform tasks with fewer distractions.

How Structured Documentation Benefits Users

Users find information faster.

Users read only what they need to complete a task.

Users do not sift through long chapters.

Information remains consistent across all manuals.

How Structured Documentation Benefits Technical Writers

Reduces duplication of content.

Enables easier collaboration.

Improves consistency in terminology and structure.

Supports content reuse.

Key Definitions / Acronyms

Core Writing Concepts

Concept

Definition A concept explains what something is, how it works, or why it matters. It provides the background knowledge a user needs before performing tasks or understanding system behavior. Concepts present information, not actions.
Purpose Build user understanding.Clarify system principles, components, relationships, or constraints.Provide foundational knowledge required before tasks or procedures.
Characteristics Non-procedural (no steps or actions).Uses clear, descriptive language.Uses noun-based headings
Examples User Roles and PermissionsSystem RequirementsSafetyIntroductionSystem Overview
When to Use When users need to learn about something before doing something.When introducing new products, features, technologies, or workflows.

Task

Definition A task is a procedural topic that explains how to do something. It contains a set of steps written in imperative commands.
Purpose Direct users through actions with clear outcomes.Support installation, configuration, operation, troubleshooting, and maintenance.
Characteristics Uses imperative verbs (for example, install, remove, connect).Uses numbered steps with one action per step.Includes prerequisites, tools, warnings, cautions, and expected results.
Examples Filter Replacement ProcedureSoftware Update Procedure
When to Use Whenever the user must perform an action.For any operation that has a sequential or procedural nature.

Reference

Definition A reference provides factual, structured information that users look up rather than read sequentially. Reference content supports informed decisions and quick access to technical details.
Purpose Present data, specifications, lists, tables, parameters, error codes, or configuration details.Give precise information without explanations or steps.
Characteristics Uses tables, lists, diagrams, or parameter lists.Avoids narrative paragraphs unless needed for clarity.Contains no procedures or step-based instructions.
Examples Technical Specifications TableError Codes and Corrective ActionsTorque ValuesConnector PinoutsSpare Parts ListUI Element Descriptions
When to Use When users need fast access to exact information.When documenting constant values, rules, limits, definitions, or options.

Active Voice

Definition A sentence structure in which the subject performs the action.
Why It Matters? It improves clarity, reduces word count, and supports an instructional tone.
Use Prefer active voice in instructions, descriptions, and warnings.
Example Tighten the connector.
Avoid The connector should be tightened.

Passive Voice

Definition A sentence structure in which the action happens to the subject.
Use Use passive voice only when the actor is unknown or irrelevant, or when the object needs emphasis.
Example The unit is calibrated at the factory.

Imperative Command

Definition A direct command form used for steps in procedures.
Use Start each step with a verb in base form.
Examples Install the driver.Replace the filter.Click OK.Click the Start button.
Avoid You should press OK.Pressing OK will…

Procedure

Definition A numbered sequence of steps that enables a user to complete a task.
Use Include prerequisites, tools, and expected outcomes. Use one action per step when possible. If additional explanation or results are needed, present them in paragraph format.
Examples Disconnect power.Remove the front cover.Install the module.See .

Step and Substep

Definition A step is a primary action; a substep breaks a step into smaller actions.
Use Use substeps when an action requires ordered micro-actions.
Examples Remove the cover.Remove the two M3 screws.Lift the cover upward.See .

Prerequisite

Definition A condition, configuration, tool, or permission required before a procedure starts.
Use Place prerequisites at the start of the procedure.

Outcome or Results

Definition The expected state after a step or procedure is completed.
Use Provide expected results to confirm success or to guide troubleshooting.
Examples The status LED remains solid green.The Window Box screen appears.

Noun Based Heading

Definition A heading that starts with a noun or noun phrase to improve consistency.
Use System OverviewPower RequirementsNetwork ConfigurationsFilter Replacement
Example Applicable for concept topics.
Avoid Verb-leading forms such as:Understanding the SystemConfiguring the Network

How Concept, Task, and Reference Work Together

In structured technical documentation, these three topics form the foundation of information architecture:

Concept = Understanding

Task = Doing

Reference = Looking up

They are usually organized so that the user first understands a concept (if required), then performs a task, and uses reference details as needed for accuracy and verification.

Example workflow:

Concept: Battery System Overview

Task: Battery Replacement Procedure

Battery Specifications and Part Numbers

Organizing content in this way reduces cognitive load, improves usability, and supports structured documentation.

Cognitive Load

Cognitive load refers to the amount of mental effort required for a user to understand information, make decisions, or complete tasks. In technical documentation, cognitive load increases when the user must interpret complex explanations, hold many details in working memory, switch contexts, or fill in missing information.

Why It Matters

User manuals should reduce cognitive load so users can focus on performing tasks safely and accurately. Excessive cognitive load leads to user errors, slower task completion, and confusion. In safety-critical environments, it can create risks.

How to Reduce Cognitive Load

Use simple, direct sentences to minimize interpretation effort.

Provide one idea per sentence and one action per procedural step.

Organize content logically so users do not need to jump back and forth between sections.

Use clear visual cues, such as tables, bullet lists, and callouts, to help users scan information.

Use consistent terminology so users do not waste mental effort interpreting synonyms.

Use illustrations to replace long technical descriptions when appropriate.

Avoid unnecessary background information, especially before tasks.

Place concepts before tasks only when needed for understanding.

Example Instead of describing a mechanism in detail before teaching how to operate it, provide a short “What you need to know” concept section, then go directly to the procedure.

Good documentation reduces cognitive load by:

Presenting information in predictable patterns.

Using structured topics (concept, task, reference).

Avoiding unnecessary complexity.

Supporting “read-to-do” or “read-to-find” behavior.

Color Usage Standards

The color wheel helps writers and illustrators maintain consistent visual style across all manuals. It ensures diagrams, callouts, UI labels, and warnings follow a predictable and accessible color scheme.

Use colors and shapes for callouts consistently throughout the document.

For example, if you use a colored callout box for UI labels, continue using that style for all UI references.

If you use a balloon with an arrow to identify hardware components, keep using that same style for all component callouts.

You may mix different shapes for different purposes, but each type must be used consistently.

Bullet Lists

Use bullets for non-sequential or unordered information. Bullets help reduce cognitive load and make information easier to scan.

Use a simple round bullet (•) for the first level.

Use a clear secondary bullet (○) for the second level.

Keep bullet text short and consistent in structure.

Avoid ending bullets with periods unless the bullet is a complete sentence.

Limit bullet lists to a maximum of two levels where possible.

Bullets may appear within a numbered step but must still follow the two-level maximum.

Example:

First-level bullet

Second-level bullet

Step Numbering

Use numbered lists for procedures. Include substeps when necessary.

Example:

Remove the cover.

Remove the screws.

Lift the panel.

Figure and Table Numbering

Number figures and tables consistently throughout the document to support clear reference and navigation.

Captions Usage Rules

Image captions apply to concept topics only.

Captions may be used when explaining system features, component identification, architecture, or reference information.

Do not use captions within procedures.

Images that appear inside task-oriented content (procedures) must not include captions because they:

interrupt step flow, and

create redundancy with the step text.

Instead reference the images directly inside the step text using simple statements such as:

See the image below.

Refer to the illustration on the right.

Writing Principles

Use Active Voice

Use passive voice only when it doesn’t matter who performs the action or when the person doing the action is unknown.

Avoid Gerunds (-ing forms)

Rewrite sentences to eliminate gerunds when possible.

Use verbs in their base (imperative) form.

Examples:

Prefer: Press the button.

Avoid: Pressing the button triggers the system.

Use Imperative Commands for Procedures

Use direct commands for procedural steps.

Examples:

Install the bracket.

Connect the cable.

Verify the connection.

Avoid auxiliary verbs: may, might, could, should in steps.

Preferred Voice for Addressing Users

Use neutral, third-person terms such as users, operators, technicians or other role-based nouns instead of second person (you).

In procedural steps, use imperative verbs; the subject is understood and does not need to be written.

Examples:

Replace the filter.

Install the module.

Verify the alignment.

Grammar and Style Rules

Terminology

Ensure consistent terminology across all manuals.

Use approved terms from the .

Use one term for one concept.

Example:

Use pushbutton, not sometimes button and other times switch

Prefer industry-standard terminology over internal jargon.

Define uncommon or highly technical terms at their first occurrence.

UI actions:

Use Click for clickable UI elements (buttons, icons).

Use Select for choosing an option from a dropdown or list.

Format UI labels in bold, matching the exact spelling and casing on the HMI or UI.

Hardware buttons:

Use Press for physical hardware buttons.

Format hardware button labels in ALL CAPS and not bold.

Example:

Press START.

Use the label exactly as printed on the device or hardware panel.

Notes to Align with STE Principles

Prefer short, concrete verbs in the imperative for procedures.

Example:

Press, Click, Select Install, Connect, Tighten, Measure.

Avoid phrasal verbs when a single verb is clearer.

Example:

Turn off switch off is fine in user manuals but in some contexts prefer de-energize for safety.

Measurement and Units

Present technical data precisely and consistently. Use standard SI and non-SI units according to engineering conventions.

Spacing Rules

Always place a non-breaking space between a number and a unit symbol. A non-breaking space (keyboard shortcut: Ctrl + Shift + spacebar) appears as a degree-like circle between words when you show paragraph marks .

Examples:

Correct: 5 mm, 12 V 20 °C

Incorrect: 5mm, 12V, 20°C

Note: Do not add space before percentage symbols.

Correct: 5%

Incorrect: 5 %

Capitalization

Unit symbols are written in lowercase, except where SI rules require uppercase (A, V, W, Pa, K, °C, Hz).

Example:

A (ampere), V (volt), W (watts), Pa (pascal), K (kelvin)

Unit names (For example, watt, pascal, volt) are written in lowercase unless beginning a sentence.

Plural Rules

Unit symbols never take plural forms.

Examples:

Correct: 5 kg, 3 lb, 10 oz

Incorrect: 5 kgs, 3 lbs, 10 ozs

Consistency Rules

Use one unit system per specification when possible (avoid mixing SI + imperial unless required).

Provide ranges with an en dash and a non-breaking space before each unit.

Examples:

Correct: 5–10 mm, 2–4 V

Acronyms and Abbreviations

Acronyms and abbreviations must be used consistently and introduced correctly to maintain clarity and prevent ambiguity.

First Use

Spell out the full term the first time it appears, followed by the acronym in parentheses.

Example:

Human Machine Interface (HMI) User Interface (UI)

Graphical User Interface (GUI)

Form Plurals Correctly

Form plural acronyms without an apostrophe.

Example:

Correct: SMEs

Incorrect: SME’s

Do not use periods in acronyms

Example:

Correct: PLC

Incorrect: P.L.C.

Use Abbreviations for Time

Use the singular unit after the number.

Examples:

5 s, 10 min, 8 h

Wait 10 min for completion.

Use Acronyms Consistently Across Manuals

Do not switch between variants (For example, UI Screen vs HMI Screen).

Select one tern based on product context and use it through.

Hyphen and En Dash

Hyphens and dashes must be used correctly to maintain clarity and precision in technical writing.

Hyphen

Use a hyphen (-) to join words that function together as a compound modifier before a noun.

Examples:

High-accuracy sensor

Two-step process

Safety-critical function

User-define setting

Hyphens prevent misreading by making the modifier act as a single unit.

En Dash

Use an en dash (–) to indicate ranges or connections between values (10–20 V). Keyboard Shortcut: Press Windows key + minus (-) to insert an en dash.

Em Dash

Avoid using em dashes in technical documentation because they make sentences longer and harder to read. If a sentence requires an em dash, rewrite it for clarity instead.

Not Recommended: The motor—when overloaded—may shut down unexpectedly.

Better: The motor may shut down unexpectedly when overloaded.

Articles

Articles must always be followed by a noun or noun phrase.

Definite Article

Use “the + singular noun” when referring to one specific, identifiable part.

Replace the cover.

Remove the bracket from the housing.

Use “the + plural noun” when referring to a specific, known group of items.

Tighten the screws to 2 N·m.

Plural Nouns without Articles

Use plural nouns without articles when referring to items in a general sense, not specific instances.

Replace covers on all units.

Inspect cables before powering on.

UI and Software Article Rules

When referencing UI elements:

Use an article when referring to the noun phrase.

Click the Reset button.

Select the Language option.

Open the Settings menu

Do not use an article when referencing the UI label itself.

Click Start.

Open Settings.

Navigate to File Save as.

This matches UI-writing standards: a UI label is treated as a proper noun, while a button/option is a noun phrase requiring an article.

General Rules

Articles are optional in imperative steps but must be used when required for clarity.

Keep article usage consistent within the same topic.

Cross references

Cross references help users find related content quickly and reliably.

Use references with labels and number when pointing to specific items within the same document.

Examples:

See Figure 3, See Section 6.4, See Table 2, See page 12

Capitalize Figure, Table, and Section when referencing a specific numbered item.

Use “See” for short, internal references.

Use “Refer to” for external or extended references.

Manual and publication titles: Italic

Example:

Refer to the General Operating Manual for more information.

Refer to the ZIRCONIA MILLING CELL MAINTENANCE MANUAL (full title) for more information.

Punctuation

Serial Comma

Use the serial comma in lists of three or more items.

Periods in Complete Sentences and Paragraphs

Use a period at the end of every complete sentence or full paragraph in running text.

Periods in Tables

Do not use periods in table cells if the content is:

a phrase

a single word

a label

a list item

Use a period only if the table cell contains a full sentence.

Periods in Bulleted and Numbered Lists

No period at the end of a bullet unless the bullet is a complete sentence.

If bullets are fragments, omit the period.

If bullets are full sentences, use a period consistently for all bullets in that group.

Example – No periods (fragments):

Mount the bracket

Install the screws

Connect power

Example – Periods (full sentences):

Mount the bracket to the enclosure.

Install the screws in the marked positions.

Periods in Steps

Use periods consistently within procedural steps. Choose either:

Imperative fragments without periods,

or

Full sentences with periods.

Whichever you choose, apply it consistently for all steps in the same procedure.

Summary Checklist

Rule Area Check
Serial comma ☐ All lists of ≥3 items use a serial comma.
Periods (sentences) ☐ Periods used for complete sentences/paragraphs.
Tables ☐ No periods in table cells unless they contain full sentences.
Bulleted lists ☐ Periods only when bullets are full sentences.
Steps ☐ Consistent use of periods (full sentences) or no periods (fragments).
Footers ☐ Period only if the footer is a complete sentence.

Best Practices for Technical Writers

Write semantically. Choose tags/styles based on meaning, not formatting.

Keep topics short and focused. One purpose per topic: Concept = what, Task = how, Reference = facts.

Avoid bold/italic/all caps unless conveying real meaning, not style.

Write step instructions as actions. Start each step with an imperative verb.
